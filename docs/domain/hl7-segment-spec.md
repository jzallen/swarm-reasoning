# HL7v2 Segment Specification — hl7-agent-factchecker

**Version:** 0.1.0
**HL7 Version:** 2.5
**Message Event:** ORU^R01 (Unsolicited Observation Result)

This document is the project's HL7v2 implementation guide. It defines
which segments are used, which fields carry meaning, which fields are
ignored, and the conventions that govern agent-generated messages.
All agents must conform to this spec. The tool layer enforces conformance
at write time by validating against this document's rules.

---

## 1. Message Structure

Every inter-agent message follows this segment order:

```
MSH          — Message header (exactly one, always first)
PID          — Claim identity (exactly one, follows MSH)
OBX[1..N]    — Observations (one or more, in sequence order)
NTE[0..M]    — Notes (zero or more, follow the OBX they annotate)
```

A complete minimal message:

```
MSH|^~\&|ingestion-agent|factchecker|orchestrator|factchecker|20260406120000||ORU^R01|RUN001-MSG001|P|2.5
PID|||CLAIM-20260406-001^^^factchecker||BidenVaccineMandate^ClaimText||20260406
OBX|1|ST|CLAIM_TEXT^^FCK||Biden issued a federal vaccine mandate for all private employers.||||||F
```

---

## 2. MSH — Message Header Segment

MSH is generated by the tool layer. Agents do not construct MSH directly.

| Field | Position | Usage | Notes |
|---|---|---|---|
| Field Separator | MSH.1 | `\|` | Fixed |
| Encoding Characters | MSH.2 | `^~\&` | Fixed |
| Sending Application | MSH.3 | Agent bundle name | e.g. `ingestion-agent`, `coverage-left` |
| Sending Facility | MSH.4 | `factchecker` | Fixed for all messages |
| Receiving Application | MSH.5 | Target bundle name or `orchestrator` | Set by orchestrator at routing time |
| Receiving Facility | MSH.6 | `factchecker` | Fixed |
| Date/Time | MSH.7 | `YYYYMMDDHHMMSS` | UTC, set by tool layer |
| Message Type | MSH.9 | `ORU^R01` | Fixed |
| Message Control ID | MSH.10 | `{run_id}-MSG{seq}` | Unique per message within a run |
| Processing ID | MSH.11 | `P` | Production |
| Version ID | MSH.12 | `2.5` | Fixed |

MSH fields not listed above are left empty.

---

## 3. PID — Claim Identity Segment

PID carries the identity of the claim under investigation. It maps the
HL7v2 patient identity concept to a claim entity.

| Field | Position | Usage | Notes |
|---|---|---|---|
| Set ID | PID.1 | `1` | Always 1 |
| Patient ID | PID.3 | `{claim_id}^^^factchecker` | Claim UUID, assigning authority `factchecker` |
| Patient Name | PID.5 | `{claim_slug}^{short_description}` | Slug is URL-safe, description is human-readable |
| Date/Time of Birth | PID.7 | `YYYYMMDD` | Claim publication or ingestion date |

PID fields not listed above are left empty. The "patient" in this system
is the claim. PID.5 family name component carries the claim slug;
PID.5 given name component carries a short human-readable description.

Example:
```
PID|||CLAIM-20260406-001^^^factchecker||BidenVaccineMandate^OSHA vaccine mandate claim||20211104
```

## 4. OBX — Observation Segment

OBX is the primary data-carrying segment. Each agent writes one or more
OBX rows representing its findings. OBX rows are append-only and
sequentially numbered within a message.

| Field | Position | Usage | Notes |
|---|---|---|---|
| Set ID | OBX.1 | Sequential integer 1..N | Assigned by tool layer, never reused |
| Value Type | OBX.2 | `ST`, `NM`, `CWE`, `TX` | See value type table below |
| Observation Identifier | OBX.3 | `{code}^^FCK` | Code from obx-code-registry.json, system `FCK` |
| Observation Value | OBX.5 | Typed value | See value type table |
| Units | OBX.6 | Unit string or empty | e.g. `%`, `score`, `count` |
| References Range | OBX.7 | `{low}-{high}` or empty | Expected range for numeric values |
| Observation Result Status | OBX.11 | `P`, `F`, `C`, `X` | See Section 5 |
| Date/Time of Observation | OBX.14 | `YYYYMMDDHHMMSS` | UTC, set by tool layer |
| Responsible Observer | OBX.16 | Agent bundle name | Which agent wrote this row |
| Observation Method | OBX.17 | Tool name or empty | Which MCP tool produced this observation |

OBX fields not listed above are left empty.

### 4.1 Value Types

| HL7v2 Type | Used For | Example |
|---|---|---|
| `ST` | Short string values ≤ 200 chars | Claim text, source name, verdict label |
| `NM` | Numeric values | Confidence score, article count, bias score |
| `TX` | Long text ≥ 200 chars | Narrative summary, full claim context |
| `CWE` | Coded values with display text | `TRUE^True^POLITIFACT`, `FALSE^False^POLITIFACT` |

### 4.2 Observation Identifier Format

OBX.3 uses a three-component coded element:
```
{code}^^FCK
```

- Component 1: observation code from `obx-code-registry.json`
- Component 2: empty (no alternate identifier)
- Component 3: `FCK` — the project's local coding system identifier

Example:
```
OBX|5|NM|CONFIDENCE_SCORE^^FCK||0.84|score|0.0-1.0||||F
```

### 4.3 Correction Pattern

When an agent corrects a prior observation, it emits a new OBX row with
`C` status. The corrected row carries the same observation code. The
tool layer sets OBX.1 to the next sequential integer. The original row
is not modified.

```
OBX|7|NM|CONFIDENCE_SCORE^^FCK||0.84|score|0.0-1.0||||F
...
OBX|23|NM|CONFIDENCE_SCORE^^FCK||0.61|score|0.0-1.0||||C
```

OBX.23 supersedes OBX.7. The synthesizer resolves by selecting the
most recent `F` or `C` row for each code.

## 5. OBX Result Status (OBX.11) Semantics

| Value | Label | Meaning | Synthesizer Behaviour |
|---|---|---|---|
| `P` | Preliminary | Hypothesis or initial finding, not yet corroborated | Informational only, not used in final verdict |
| `F` | Final | Agent considers this finding settled | Authoritative input to synthesis |
| `C` | Corrected | Supersedes an earlier observation of the same code | Authoritative input, replaces prior `F` |
| `X` | Cancelled | Claim not check-worthy or finding retracted | Excluded from synthesis, retained for audit |

An agent may emit `P` rows freely during its reasoning process and
must emit at least one `F` or `X` row for each of its registered codes
before the orchestrator considers it complete (see ADR-005, ADR-010).

## 6. NTE — Notes and Comments Segment

NTE segments carry unstructured text that annotates the immediately
preceding OBX row. They are optional and do not carry epistemic status.

| Field | Position | Usage |
|---|---|---|
| Set ID | NTE.1 | Sequential integer |
| Source of Comment | NTE.2 | Empty (not used) |
| Comment | NTE.3 | Free text, UTF-8, ≤ 512 chars |

Example:
```
OBX|12|NM|BLINDSPOT_SCORE^^FCK||0.78|score|0.0-1.0||||F
NTE|1||Right-spectrum sources silent on this claim. Only 2 of 14 sources are center or right.
```

## 7. ACK — Acknowledgment Message

When a subagent's Mirth instance receives a message, it returns an ACK.
The orchestrator monitors ACKs to track delivery. ACK is not a completion
signal — completion is determined by OBX terminal status (see ADR-010).

```
MSH|^~\&|coverage-left|factchecker|orchestrator|factchecker|20260406120005||ACK|ACK001|P|2.5
MSA|AA|RUN001-MSG003|Message received successfully
```

| MSA.1 Value | Meaning |
|---|---|
| `AA` | Application Accept — message received and stored |
| `AE` | Application Error — message received but could not be processed |
| `AR` | Application Reject — message rejected, do not resend |

`AE` triggers a retry by the orchestrator (up to 3 attempts). `AR`
triggers an escalation to the run error log.

## 8. Escape Sequences

HL7v2 field delimiter characters must be escaped when they appear in
field values. The tool layer applies escaping automatically. This section
documents the convention for reference.

| Character | Escape Sequence | Context |
|---|---|---|
| `\|` (field separator) | `\F\` | In any field value |
| `^` (component separator) | `\S\` | In any field value |
| `~` (repetition separator) | `\R\` | In any field value |
| `\` (escape character) | `\E\` | In any field value |
| `&` (subcomponent separator) | `\T\` | In any field value |

Example — claim text containing a pipe character:
```
OBX|1|ST|CLAIM_TEXT^^FCK||Congress passed the bill \F\ the president signed it.||||||F
```

## 9. Verdict Mapping to PolitiFact Scale

The synthesizer emits a `VERDICT` OBX row of type `CWE`. The coded
value maps to PolitiFact's six-tier scale as follows:

| OBX.5 Coded Value | PolitiFact Equivalent | Confidence Range |
|---|---|---|
| `TRUE^True^POLITIFACT` | True | 0.85 – 1.00 |
| `MOSTLY_TRUE^Mostly True^POLITIFACT` | Mostly True | 0.70 – 0.84 |
| `HALF_TRUE^Half True^POLITIFACT` | Half True | 0.45 – 0.69 |
| `MOSTLY_FALSE^Mostly False^POLITIFACT` | Mostly False | 0.25 – 0.44 |
| `FALSE^False^POLITIFACT` | False | 0.10 – 0.24 |
| `PANTS_FIRE^Pants on Fire^POLITIFACT` | Pants on Fire | 0.00 – 0.09 |
| `UNVERIFIABLE^Unverifiable^FCK` | No PolitiFact equivalent | N/A |

The `CONFIDENCE_SCORE` OBX row (NM type, 0.0–1.0) drives the verdict
mapping. The synthesizer emits both rows together as `F` status.
`UNVERIFIABLE` is used when no primary sources or ClaimReview entries
exist and coverage analysis is insufficient for a confidence score.

## 10. Message File Conventions

- Messages are stored as `.hl7` files in YottaDB-backed storage.
- File naming: `{run_id}.hl7`
- Segment terminator: carriage return `\r` (0x0D), per HL7v2 standard.
- Character encoding: UTF-8.
- MLLP framing (0x0B / 0x1C 0x0D) is used only for TCP transport via
  Mirth. Files written to YottaDB do not include MLLP framing bytes.
